# Repository Guidelines

## 项目目标
本项目的目标是将英文电子书词汇分析能力产品化。用户上传 `epub` 后，系统提取单词、统计频率、补充 `COCA` 排名与原型信息，并结合“已掌握词”生成 3 份 CSV：完整单词列表、待记忆单词列表、95% 覆盖待记忆单词列表。

## 目标结构
- `backend/`：`FastAPI` 服务、异步任务、数据库模型、文件存储与 CSV 导出。
- `frontend/`：`Nuxt` 产品前端，包含首页、个人词库、登录注册弹框、书架与历史记录页。
- `docs/`：接口约定、CSV 字段说明、业务规则。
- `scripts/`：迁移或开发辅助脚本，不承载正式业务。

原始书籍、用户词表、生成 CSV 不应长期存放在仓库中，应保存到服务器文件存储或对象存储；数据库仅保存元数据、任务状态、结果索引和统计信息。

## 核心业务规则
- 已掌握词 = 下拉框选择的 `COCA 1000..15000` 词汇，与用户上传个人词表的并集。
- 分析任务必须为异步模式：上传后返回 `job id`，前端轮询或订阅状态。
- 支持同一本书按不同已掌握词范围重复分析。
- 95% 覆盖率定义为：待记忆词按书中出现频率累计后，对全书总词频达到 `95%`。

## CSV 规范
三类 CSV 都必须包含以下字段：`序号`、`单词`、`书籍出现频率`、`COCA 排行`、`原型`、`是否已掌握`。未经同步更新导出逻辑、接口文档和前端展示，不得擅自修改列名、列顺序或字段语义。

## 编码规范
- Python 使用 4 空格缩进，前端代码使用 2 空格。
- Vue 使用组件化、工程化结构；禁止把复杂页面逻辑全部堆在单个页面文件。
- 注释使用中文；变量名、函数名、模型名、接口字段名使用英文技术词。
- 新增后端接口必须定义清晰的请求模型与响应模型。

## 数据与数据库
数据库设计应按产品级标准扩展，至少覆盖：`users`、`user_vocabularies`、`user_vocabulary_items`、`books`、`analysis_jobs`、`analysis_results`、`analysis_result_items`。用户词表上传格式暂定为 `CSV`。

## 禁止事项
- 不直接修改词库源数据格式，除非同步更新解析逻辑与文档。
- 不提交 `node_modules`、临时文件、导出产物或测试生成文件。
- 不把实验性脚本混入正式业务流程。
- 不在变更过滤逻辑、原型逻辑、覆盖率逻辑后遗漏文档更新。
